### 一、后端重启了或前端变更代码了，是否要重启前端

#### 前端变更代码
  需要重启的情况 ：
- 更改了 Vite 配置文件（vite.config.js）
- 安装了新的依赖包（需要 npm install 后重启）
- 遇到构建缓存问题（偶尔需要手动重启清理缓存）
#### 后端重启
前端的处理 ：
- 不需要重启前端开发服务器 ，但可能需要：
    1. 刷新页面 ：后端重启后，前端的连接会断开，需要刷新页面
    2. 重新登录 ：如果后端使用了 session 或 token 机制，重启后可能需要重新登录
    3. 清除缓存 ：如果前端有本地存储的缓存，可能需要清除



### 二、标准开发流程参考：

**先定义数据库表** → 生成 **PO**。

**定义 Service 接口**（思考业务需要什么功能，先定契约）。

**定义 Controller 层的 Request 和 VO**（思考前端要传什么、返回什么）。

**编写 Mapper/DAO**（满足 Service 需要的数据查询）。

**编写 ServiceImpl**（实现核心逻辑，调用 Mapper，进行 PO ↔ DTO 转换）。

**编写 Controller**（调用 Service，进行 DTO ↔ VO 转换，返回结果）。



### 三、对象职责划分

| 对象类型            | 位置          | 作用         | 说明                                                         |
| ------------------- | ------------- | ------------ | ------------------------------------------------------------ |
| **PO**              | Dao 层        | 数据库实体   | 与数据库表一一对应，里面只有 getter/setter，不掺业务逻辑。   |
| **DTO**             | Service 层    | 数据传输对象 | 用于 Service 内部或 Service 与 Controller 之间传数据。比如你要联合查询 PO 和其他数据，封装成 DTO。 |
| **VO**              | Controller 层 | 视图对象     | 返回给前端的 JSON 结构。字段名和类型完全由前端需求决定（和 PO 可能不一样）。 |
| **Request / Query** | Controller 层 | 请求参数     | 接收前端传来的参数（你已经写了）。                           |

### 五、ID参数的传递，前端传什么类型给后端好

> 对于ID参数的传递，建议前端使用 String类型 传递，后端再转换为Long类型处理

1.JavaScript数字精度问题 ：

- JavaScript的Number类型是64位浮点数，对于超过16位的整数会丢失精度
- Long类型的ID（如自增主键）可能超过这个范围，导致前端接收到的ID与后端不一致

2.兼容性更好 ：

- String类型可以表示任意长度的数字，不会有精度损失
- 避免了类型转换的潜在问题

3.后端处理简单 ：

- 后端只需要一行代码转换： Long.parseLong(idString)

##### 线上业务库（MySQL）：绝对禁止「直接物理 DELETE」

所有面向用户 / 后台的删除接口，**统一只做逻辑删除**

原因：

- 大表 DELETE 会造成**锁表、主从延迟、IO 抖动**
- 误删无法回滚，业务事故等级极高
- 无法做操作审计、数据溯源
- 关联删除（如你的 MySQL + Milvus）无法保证一致性

##### 主流标准流程

1. **业务删除 → 软删标记**（毫秒级，无风险）
2. **保留窗口期**（7/30/90 天可恢复）
3. **离线低峰定时任务**（凌晨 2~4 点）
4. **批量物理删除 / 归档历史表**
5. **关联中间件（Milvus/ES/Redis）同步清理**

### computeIfAbsent方法详解

```java
    default V computeIfAbsent(K key,
            Function<? super K, ? extends V> mappingFunction) {
        Objects.requireNonNull(mappingFunction);
        V v;
        if ((v = get(key)) == null) {
            V newValue;
            if ((newValue = mappingFunction.apply(key)) != null) {
                put(key, newValue);
                return newValue;
            }
        }

        return v;
    }
```

**参数：**Key、方法

**核心原理：**

1.根据Key获取value，若value为null

2.构造返回值对象

3.调用方法传入Key

4.赋值新对象返回

### Query重写产生子问题之后，进行意图识别，这时候对象是谁？原问题？子问题？all？

**意图识别的对象 = 所有子问题**

原问题、重写后的母问题，**都不参与意图识别**。

### 为什么是这样

1. 母问题（原 Query / 重写 Query）通常是**复合意图**，意图模型很难直接识别出单一、可执行的意图。
2. 子问题已经是**原子化、单意图**的查询，正好匹配意图识别的输入要求。
3. 意图识别的目的是为了**路由 / 检索 / 调用工具**，这些都只针对子问题执行，不需要再看母问题。

### 标准流程

原 Query → Query 重写 → 拆分子问题 → **对每个子问题分别做意图识别** → 分别检索 / 调用工具 → 合并答案





### 在RAG场景中，子问题的数量跟最大意图情况的数量如何配置

召回率（Recall）” vs “精确率（Precision）

**子问题数量（N）：** 决定了 “检索的广度”。

- **过少**：复杂问题拆解不充分，遗漏信息点。
- **过多**：引入噪音，稀释核心问题，且增加 Latency（延迟）。

**最大意图 / Collection 数量（M）：** 决定了 “检索的范围”。

- **过少**：信息茧房，跨领域知识无法召回。
- **过多**：检索太杂，引入不相关文档，增加 Rerank 负担。



主流方案通常遵循：N >= M

### 用父类型接收子类型对象，这是什么特性why？what？

**是什么：**

- **面向接口编程**（这是设计思想）
- **向上转型**（这是语法本质）
- **多态性应用**（这是底层特性）

**标准描述：**

- 使用**父接口（Collection）** 作为引用类型，接收其**实现类（ArrayList/HashSet/LinkedList）** 的实例对象。

**优势：**

- **解耦**：不依赖具体实现类，后续想换成 `LinkedList`/`HashSet`，只改 `new` 的部分，其他代码不用动；
- **通用**：符合 Java 设计规范，方法传参 / 返回值优先用父接口；
- **灵活**：依托多态，运行时自动调用子类的实现方法。





## 召回结果进行Rerank时，召回文本的得分会参考吗？

答案是：**召回分数重排模型完全不参考**：重排模型的输入只有 `query` 和 `documents` 纯文本列表，它是**独立重新计算语义相关性**的，召回阶段的向量相似度分数对它没有任何影响，也无法传入。

**底层重排模型 / SDK 没有最小分数限制**：官方 API 和 SDK 都没有提供 `min_score` 这类参数，只会按 `topN` 做数量截断，不管分数多低都会返回前 N 个。

**重排后的分数是否要再限制，看你的业务场景**：严谨场景（法律、医疗、企业知识库）**强烈建议加分数过滤**，避免把低质量 / 不相关的文档塞给大模型；通用闲聊场景可以不加。
